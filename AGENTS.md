## Universal conventions

These rules apply to every session. Each is grounded in a failure
that happened (or nearly happened) in session 1.

### Working with the GitHub MCP server

#### MCP returns body content HTML-escaped

**Do**: always run `html.unescape()` on issue/comment bodies before
JSON-parsing or literal-string matching.

**Why**: MCP returns `&#34;` instead of `"`, etc. Strict parsers fail
silently. Idempotent on REST-returned content, so safe to apply
unconditionally.

**See**: `retrospective/github-mcp-tips/SPEC.md` §2.1

#### MCP-posted comments get an auto-trailer

**Do**: parse posted comments with
`json.JSONDecoder().raw_decode(body.lstrip())`, not strict
`json.loads()`.

**Why**: `mcp__github__add_issue_comment` appends
`\n\n---\n_Generated by [Claude Code](https://claude.ai/code)_`
to every comment. Strict JSON parsing fails on the trailer.

**See**: `retrospective/github-mcp-tips/SPEC.md` §2.2

#### Don't lock issues that workflows need to write to

**Don't**: gate workflows on `github.event.issue.locked == true`.

**Why**: Locked issues refuse comments from `GITHUB_TOKEN`, even with
`permissions: issues: write`. The workflow's own writes get 403. This
was the highest-impact bug in session 1.

**Do**: filter on label + author. Lock at issue **close**, not at
creation, if you need a tamper-prevention seal.

**See**: SPEC.md §3 "Real-world correction";
`retrospective/github-mcp-tips/SPEC.md` §2.7

#### Source `agent_login` from `mcp__github__get_me`

**Don't**: hardcode `agent_login` to a placeholder like `"my-bot"`.

**Why**: The MCP server authenticates as the human user (e.g.
`jonathanmanton`), not as a bot. Workflow `if:` filters comparing to
a wrong identity will never match.

**Do**: At session start, fetch the actual login via
`mcp__github__get_me` and pass it to skill scripts (`submit.py`,
`lock_and_sweep.py`) as the explicit `agent_login` argument. The static
`agent_login` config key was removed in session 3. CI workflows source
the same value from a repo-level `vars.AGENT_LOGIN` GitHub Actions
variable; see SPEC §3.1.

#### Don't expect to read CI logs

**Note**: `mcp__github__*` tools do not expose workflow run logs.
WebFetch on `/actions/runs/<id>` hits an auth wall.

**Do**: workflows already include marker-comment + base64-embedded
stdout patterns. If something fails, look at the
`<!-- workflow-marker -->` comments on the issue. See
`retrospective/live-debug-from-mcp-only/SPEC.md` for the full pattern.

### Branch and PR coordination

#### Subagent branch naming

**Do**: use `<feature>--sub-<id>` (double-dash separator).

**Don't**: use `<feature>/sub-<id>` (single-slash). Git refs use the
filesystem; `agent/foo` and `agent/foo/bar` cannot coexist.

**See**: `retrospective/github-mcp-tips/excerpts.jsonl` →
`git-ref-prefix-collision`

#### Don't force-push during an open PR

**Don't**: `git push -f origin <feature-branch>` while a PR from
`<feature-branch>` is still open.

**Why**: If the force-push aligns the branch with main, GitHub
auto-closes the PR (head == base, no diff). Lost PR #6 this way in
session 1; recovered from local reflog.

**Do**: After PR merge, sync via:
```bash
git fetch origin
git reset --hard origin/main
git push -f origin <branch>
```
ONLY after the PR is merged.

#### Cleanup happens automatically

`close_on_merge.py` sweeps the PR's head branch + any
`<feature>--sub-*` siblings on every PR merge. Defensive: never
touches `main`, `_agent_runs`, or anything outside the `agent/`
namespace.

To delete a branch outside that flow, you'll need to drive deletion
through the system itself (push empty commit, open PR, squash-merge),
because the local git proxy in this sandbox blocks
`git push --delete`.

**See**: `retrospective/forensic-vs-aggressive-cleanup/SPEC.md`

### Subagent dispatch

#### Briefs must specify exact deliverables

Every subagent brief ends with:
1. The exact `git commit` + `git push` commands
2. The exact `mcp__github__create_pull_request` invocation (title +
   base + head + body shape)
3. The report format ("PR number, file list, test count delta, any
   caveats — under N words")

**Why**: Without this, results vary wildly. Reports balloon, branches
don't get pushed, PRs don't get opened.

**See**: `retrospective/subagent-prompting/SPEC.md` §2

#### Subagents must commit locally before stopping

**Do**: in subagent briefs, instruct: "if you commit local changes
during your work, push them before stopping. The dispatcher can pick
up local-but-unpushed work."

**Why**: Subagents can run out of usage mid-task. If they've
committed locally, recovery is trivial. Without a local commit, the
work is lost.

**See**: `retrospective/subagent-prompting/SPEC.md` §6;
example of recovery in session 1's lock-vs-bot fix.

#### Briefs should include "don'ts"

Every brief includes "don't refactor unrelated code", "don't add
features beyond this brief", "don't break existing tests" plus
task-specific don'ts.

**Why**: Subagents will scope-creep without explicit constraints.

### Workflow scripts

#### `__main__` blocks must validate env vars

```python
def main() -> int:
    required = ["ISSUE_NUMBER", "COMMENT_ID", "GITHUB_TOKEN", "GITHUB_REPOSITORY"]
    missing = [n for n in required if not os.environ.get(n)]
    if missing:
        print(f"missing env vars: {missing}", file=sys.stderr)
        return 1
    # ... rest
```

**Why**: makes "how do I invoke this" self-documenting; catches drift
between YAML and code.

#### Diagnostic comments are first-class

Workflow scripts post a self-diagnostic comment on uncaught
exception. Already implemented in `handler.py`, `lock_and_sweep.py`,
`close_on_merge.py`.

When adding a new workflow script, copy the pattern. See
`retrospective/live-debug-from-mcp-only/SPEC.md`.

### Spec discipline

#### Run live scenarios after unit tests pass

**Don't**: ship based on unit-test green. They only prove the impl
matches your understanding of the spec. The spec might be wrong.

**Do**: run live scenarios that exercise architecturally-distinct
code paths. Spec defects and transport quirks only surface live.

**See**: `retrospective/spec-vs-implementation-gap-discovery/SPEC.md`

#### Annotate spec amendments inline

When live execution discovers a spec problem, amend the spec in
place with:

```markdown
**Real-world correction (discovered <when> via <how>).** <One-sentence
description of what was wrong>. <One-sentence statement of the new
design>.
```

Don't move the original; correct it explicitly.

**Examples**: SPEC.md §3 (lock placement), §5.2 (comment-trailer
tolerance), §6 (branch naming).
