## Part 1 — what happened

### Phase 1 — status assessment

Started from a cold-pickup prompt ("Where are we with this project?"). Read `README.md`, `HANDOFF.md`, `retrospective/README.md`, and `retrospective-2026-05-06/README.md` to establish state. The POC was feature-complete: 446 tests passing, 11/11 live scenarios green, 9 real-world bugs fixed across sessions, no urgent open work per `HANDOFF.md`'s "Open work" section.

Initially answered the user's "is it time to make this process repeatable?" question as if they were asking about implementing the retrospective skill specs in `retrospective/` and `retrospective-2026-05-06/`. The user immediately corrected: they meant skills *for* the protocol itself — an execution engine for multi-agent work — not skills *from* retrospectives.

### Phase 2 — design conversation (~5 rounds of clarification)

After the misunderstanding, used `AskUserQuestion` four times to nail down the design:

1. Skill structure (onboarding triggers install vs separate vs both).
2. Resume model for interrupted onboarding (summarize and ask vs silently resume vs restart).
3. Adoption marker (`.agent/config.json` vs sentinel file vs settings.json entry vs combination).
4. Output scope (which files the onboarding skill may edit).

The user's answers consolidated into a tight design: each skill self-installs, onboarding is opt-in and triggered by other skills' detection, dialog file on well-known branch persists state, AGENTS.md/CLAUDE.md are sacred (pointer-line edits only).

Subsequent rounds covered: distribution form (recipe + tarball from one source), test harness model (agent's own MCP credentials, not a separate test account), template-source relationship (copies of POC source, contract-tested), repo containment (everything new under one subdirectory).

## Part 4 — proposed ADRs

Architectural decisions made in this session that the user may want to record as ADRs. **Titles only — no specs.** Use the `adr` skill to author any of these if you decide they're worth recording.

- **Distribute the agent-job protocol as a skill file package only** — no separate installer, no PyPI, no Docker; everything ships as Claude Code skill files. Grounded in the user's explicit decision when offered alternatives.
- **Project-level skill scope (`.claude/skills/`), not user-level** — keeps the skill set co-located with the protocol files (`.agent/`, workflows) it drives. Same audit boundary, same review surface.
- **Each protocol skill self-bootstraps its templates on first invocation** — no separate install step; skills carry their templates as reference files and copy them via tool calls when missing.
- **Onboarding is opt-in and never required** — the four protocol skills work standalone. Onboarding is offered when its absence is detected, but a declined onboarding does not gate the protocol skills.
- **`AGENTS.md` and `CLAUDE.md` are sacred** — the maximum-allowed change is a pointer line ("see <file>"); even that requires explicit per-file user approval. Justification: users copy these between projects.
- **Recipe Markdown + tarball, generated from one source** — two distribution forms, byte-equivalent content, produced by a single `build.py`. Manual edits to either form forbidden.
- **Bundled templates are byte-equivalent copies of POC source, contract-tested** — POC remains canonical; a test asserts byte-match between bundled template and POC source. If POC source changes, the test fails until bundle is regenerated.
- **The test harness is development-only — not part of end-user distribution** — it ships in the bootstrap bundle but the future distribution build excludes it.
- **The test harness uses the agent's own GitHub MCP credentials** — no separate test account, no PAT setup, no secrets configuration. The same dispatch model that drove the POC drives the maintenance repo's tests.
- **The new repo dogfoods the protocol it maintains** — `pipeline-ai-sandbox`'s own development goes through the very dispatch protocol the skills implement.
- **POC stays preserved as a historical reference** — derived work goes in a separate subdirectory (`pipeline-skills-package/`); future maintenance moves to the new `pipeline-ai-sandbox` repo entirely.
- **Plans are designed to be parallel-execution-ready** — `PLAN-PACKAGE.md` and `NEW-REPO-PLAN.md` follow the parallel-subagent-fanout pattern with `state.json` restart safety and single-message wave dispatch.
