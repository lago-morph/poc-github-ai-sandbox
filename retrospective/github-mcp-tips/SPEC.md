# `github-mcp-tips` — implementation spec

A reference skill that an agent loads at the start of any
GitHub-MCP-heavy task. Encodes transport quirks, tool gaps, and
workarounds discovered live.

## 1. Trigger conditions

Activate when the agent is about to use `mcp__github__*` tools
substantively:

- Posting more than ~3 comments
- Reading issue or comment bodies and parsing structured data from them
- Driving GitHub Actions workflows
- Branch / PR orchestration
- Cross-repo or multi-issue tasks

Soft-trigger (just have the skill loaded):
- Any session where the user mentions GitHub explicitly
- Any session in a repo with `.github/workflows/`

## 2. Core content (the substance)

### 2.1 Body content is HTML-escaped on read

Every issue body, comment body, and PR body returned by
`mcp__github__*_read` tools has HTML entity encoding:

```
{
  "body": "{\n  &#34;protocol_version&#34;: 1,\n  ..."
}
```

`&#34;` = `"`, `&amp;` = `&`, `&lt;` = `<`, etc.

**Always** run `html.unescape(body)` before:
- JSON parsing
- Regex matching against literal punctuation
- Comparing to a string with quotes

Idempotent on already-unescaped content (REST clients return literal
quotes), so safe to apply unconditionally.

```python
import html, json
parsed = json.loads(html.unescape(comment_body))
```

### 2.2 Posted comments get an auto-appended trailer

`mcp__github__add_issue_comment` unconditionally appends:

```
\n\n---\n_Generated by [Claude Code](https://claude.ai/code)_
```

To every comment body. This breaks strict JSON parsing.

**Always** parse posted comments with `JSONDecoder().raw_decode()`,
which returns the longest JSON-object prefix and ignores trailing
content:

```python
import json
def parse_envelope_lenient(body: str):
    body = body.lstrip()
    if not body:
        return None
    try:
        parsed, _ = json.JSONDecoder().raw_decode(body)
        return parsed
    except (json.JSONDecodeError, ValueError):
        return None
```

If you control both ends (write and read via MCP), the trailer is
always present. If you read a comment that was written via REST
(e.g., a workflow handler updating its own envelope), no trailer.
The lenient parser handles both.

### 2.3 Confirmed missing tools

As of the session date, these operations have **no MCP tool** and
must be worked around:

| Operation | Workaround |
|-----------|------------|
| Update / edit a comment | Post a follow-up comment as a "reply"; OR have the workflow side use REST. |
| Lock / unlock an issue | Have a workflow do it via REST. Agent cannot. |
| Delete a comment | None reliable. Comment is permanent. |
| Read workflow run logs | See `live-debug-from-mcp-only` skill: marker comments + base64-embedded tail. |
| Trigger `workflow_dispatch` | Push a commit that triggers `on: push`. |
| Re-run a failed workflow | Same. Or post a new comment that triggers. |
| Get a single comment by id | Use `issue_read` `get_comments` and filter. |
| Set repo Actions variables | Hardcode in YAML, or use `mcp__github__create_or_update_file` to patch. |
| Read repo Actions secrets | None — secrets are write-only via the API. |

### 2.4 Tool-name footguns

- `mcp__github__issue_read` methods: `get`, `get_comments`,
  `get_sub_issues`, `get_labels`. **No `get_comment`** (singular).
- `mcp__github__issue_write` methods: `create`, `update`. **No `lock`.**
- `mcp__github__pull_request_read` is the right tool for PR-specific
  reads (has `get_check_runs` for CI status).
- `mcp__github__add_issue_comment` works on issues AND PRs (PR
  number = issue number for comment purposes).
- Tool schemas are deferred. Use `ToolSearch` with
  `select:tool1,tool2` to load schemas before invoking, or with a
  natural-language query to find tools.

### 2.5 Branch and ref operations

- `mcp__github__create_branch` works including from arbitrary
  `from_branch`. Returns the new ref + the source SHA.
- `mcp__github__list_branches` paginates via `Link: rel="next"`. The
  MCP wraps it but the underlying REST client (if you write one)
  needs to handle the header explicitly.
- **Git ref-prefix collision is real** and caught me out:
  `agent/foo` and `agent/foo/bar` cannot both be refs simultaneously.
  Use a flat separator for hierarchical naming:
  - GOOD: `agent/foo--sub-alpha` (double-dash)
  - GOOD: `agent-sub/foo-alpha` (separate namespace)
  - BAD: `agent/foo/sub-alpha` (collides with `agent/foo`)
- Branch names must be < 256 bytes total.
- `_agent_runs` (or any orphan log branch) requires manual creation:
  blob → tree → commit-no-parents → ref. The Git Database API. The
  MCP `create_or_update_file` won't create an orphan branch on its
  own — it expects the branch to exist. Provide a code template for
  this.

### 2.6 Webhook and workflow timing

- `issue_comment.created` webhooks fire ~immediately, but the
  workflow runner takes 5-60s to pick up.
- `pull_request.closed` (with `merged: true`) fires after the merge.
- `issues.opened` fires when an issue is created.
- A typical `ubuntu-latest` workflow: 30-90s end-to-end (15s setup
  + the actual work + 5s teardown).
- Polling interval rule of thumb: 30s for the first 5 minutes, then
  60s. Hard cap polling at 1 hour for any single operation.

### 2.7 Permissions cheat-sheet

For workflow YAMLs that talk back to the repo:

```yaml
permissions:
  contents: write    # commits, branch creation, branch deletion
  issues: write      # issue + comment writes (NOT enough on locked issues)
  pull-requests: write  # PR creation/update
  actions: read      # to query workflow runs
```

Special cases:

- **Locked issues**: even with `issues: write`, `GITHUB_TOKEN` is
  refused. There is no permission that grants comment-on-locked-issue
  to the bot. Don't lock issues that the workflow needs to write to.
  If you need a tamper-prevention seal, lock at issue close.
- **Branch deletion**: requires `contents: write`. Sounds like it
  should be `contents: read` since it's metadata, but it isn't.

### 2.8 Identity / `agent_login` patterns

The MCP server authenticates as the user (e.g., `jonathanmanton`),
not as a bot. Workflow `if:` filters comparing
`comment.user.login == 'some-bot-name'` won't match the agent's
writes.

**Always** source the actual login from `mcp__github__get_me` once
at session start, persist it in a config file, and refer to that.
Don't use placeholder strings like `"my-bot"`.

```python
# At session start:
me = mcp__github__get_me()
agent_login = me["login"]  # e.g., "jonathanmanton"
# Persist to .agent/config.json or similar.
```

### 2.9 Pagination

Several tools paginate at 30 items/page by default:

- `list_branches`
- `list_comments`
- `list_issues`
- `list_pull_requests`

Pass `perPage=100` for fewer round-trips, or follow `Link: rel="next"`
through pages. The MCP wraps the latter for you in most cases.

### 2.10 Search vs list

- `mcp__github__search_*` tools have global rate limits (separate
  from REST quotas).
- For "what comments are on issue #N", use `issue_read get_comments`,
  not `search_issues`.
- For "what issues mention X", search is appropriate.

### 2.11 WebFetch as a fallback

When MCP doesn't expose what you need, `WebFetch` against
`github.com` URLs sometimes works for **public** content:

- ✅ `https://github.com/<owner>/<repo>/actions` — workflow run list
- ⚠ `https://github.com/<owner>/<repo>/actions/runs/<id>` — often hits
  "Sorry, something went wrong" (auth wall on log details)
- ❌ Anything requiring sign-in (private repos, log archives, settings)

Don't rely on WebFetch for anything deterministic. It's a
diagnostic-of-last-resort.

## 3. Skill output format

When activated, the skill should:

- Inject `html.unescape` and `JSONDecoder().raw_decode()` patterns
  into any code the agent generates that parses MCP-returned bodies.
- Steer agent away from confirmed-missing tools (no "let me try
  update_comment...").
- Recommend the diagnostic-comment pattern when the agent is about
  to debug a CI failure.
- Flag illegal branch-naming patterns at design time.
- Recommend `mcp__github__get_me` for any reference to "the bot's
  login" or "the agent's login."

## 4. Anti-patterns

- Hard-coding `agent_login` to a placeholder string (`"my-bot"` etc.).
- Strict `json.loads(comment_body)` without trailer tolerance.
- Locking an issue that the workflow needs to write back to.
- Using `/sub-` separator in subagent branch names.
- Polling workflow runs faster than once every 30s.
- Trying to read workflow logs from MCP — they're not exposed.

## 5. Implementation notes for the builder

The skill itself can be quite simple — much of its value is *just
the documented checklist*. A SKILL.md file with these sections,
loaded by the agent harness when GitHub MCP tools are about to be
used, would deliver most of the value.

Optionally:

- A small Python helper module (`github_mcp_helpers.py`) with the
  parsers (`parse_lenient_envelope`, `parse_meta_with_unescape`),
  the orphan-branch creator, the diagnostic-comment poster.
- A linter rule that flags `json.loads(<comment-body-var>)` and
  suggests `parse_lenient_envelope` instead.

## 6. Test plan (when built)

- Verify each parser against real recorded MCP responses (fixture
  bundle of known-good and known-trailered bodies).
- Verify the orphan-branch creator builds a valid first commit on a
  scratch repo.
- Round-trip test: post a comment, read it back, parse it cleanly.
