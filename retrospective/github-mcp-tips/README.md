# `github-mcp-tips` skill

> The friction points, transport quirks, tool gaps, and workarounds
> for the GitHub MCP server — captured before they're forgotten.

## Why this skill

Live execution of the agent-job-protocol against real GitHub via
`mcp__github__*` surfaced **7 distinct bugs** in roughly 90 minutes,
none of which would have appeared in unit tests. Each one took a fresh
round of investigation. A future agent that knows about them upfront
saves all of that.

Examples of real surprises this skill will encode:

- `mcp__github__add_issue_comment` unconditionally appends a
  `\n---\n_Generated by [Claude Code]_` trailer to every comment body.
  Strict `json.loads()` fails on this. Took multiple iterations to
  diagnose.
- Issue and comment bodies returned by `mcp__github__*_read` are
  **HTML-escaped** — `&#34;` instead of `"`. The protocol's JSON
  parser failed silently until `html.unescape()` was added.
- A locked GitHub issue refuses comments from `GITHUB_TOKEN`, even
  with `permissions: issues: write`. The protocol's whole design
  hinged on lock-then-write-back, which is structurally impossible.
- Branch refs like `agent/foo` and `agent/foo/bar` cannot coexist.
  Discovered when scenario 01 tried to create a sub-branch under an
  existing feature branch.
- No MCP tool exists for: editing comments, locking/unlocking issues,
  fetching workflow logs, triggering `workflow_dispatch`. Major gaps
  that need workarounds.

These are the kind of lessons that only matter once you hit them, and
matter a lot when you do.

## When this would have helped

- **Scenario 01, attempts 1-3**: I wasted four PRs (#13, #14, plus
  the diagnostic ones) figuring out why the handler was crashing. If a
  skill had said "locked issues block GITHUB_TOKEN writes," I would
  have fixed it in 5 minutes instead of 90.
- **Comment parsing**: the trailer-tolerance issue cost a full
  subagent dispatch + PR cycle. The skill could have made it a known
  fact, encoded in templates.
- **Branch naming**: the spec literally proposed an illegal pattern
  (`<feature>/sub-<id>`). The skill would have flagged this at design
  time.

## What good looks like

Future agent reads this skill at the start of any GitHub-MCP-heavy
task. Skill provides:

- A list of confirmed-missing tools (so the agent doesn't waste time
  searching for them).
- Parser/fetcher templates that already include the necessary
  `html.unescape()` and `JSONDecoder().raw_decode()` patterns.
- The "real workflow timing" expectation (30-90s for a typical
  handler run) so polling intervals are right by default.
- The "diagnostic comment" pattern (see also the
  `live-debug-from-mcp-only` skill) so debugging CI failures is
  approached correctly from the start.

## Concrete value

Estimated dispatcher time saved per future GitHub-MCP project: **2-4
hours**. Estimated subagent invocations saved: **3-6** (one per bug
that doesn't have to be rediscovered).
