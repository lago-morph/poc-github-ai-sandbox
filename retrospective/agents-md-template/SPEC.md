# `agents-md-template` — implementation spec

A spec for what to add to `AGENTS.md` (or `CLAUDE.md`) in projects
that use GitHub MCP, subagents, or live execution patterns.

## 1. Trigger conditions

- Setting up a new project that will use these patterns.
- Augmenting an existing `AGENTS.md` after a session like this one.
- Onboarding a new agent harness to an existing repo.

## 2. Output deliverable

A markdown file (`AGENTS.md` at repo root) with the conventions
below. Adapt section names / counts to the project's needs. Keep
total under ~200 lines.

The format for each rule:

```markdown
### N. <Rule name>

**Do/Don't**: <one-line statement>

**Why**: <1-2 sentences grounded in real failure>

**See also**: <link to skill or longer doc>
```

## 3. Rules to include (the conventions discovered this session)

### 3.1 Branch naming and ref-prefix collisions

**Don't**: use `<feature>/sub-<id>` as a subagent branch pattern.

**Why**: Git refs use the filesystem; you cannot have a branch at
both `agent/foo` and `agent/foo/bar` simultaneously. Discovered
when scenario 01 returned 422 on branch creation.

**Do**: use `<feature>--sub-<id>` (double-dash separator) so the
subagent branch sits at the same prefix level as the feature
branch.

**See also**: `retrospective/github-mcp-tips/SPEC.md` §2.5

### 3.2 PR + branch coordination

**Don't**: force-push a feature branch back to main's tip while a
PR from that branch is still open.

**Why**: GitHub auto-closes the PR (head == base, no diff). Lost
PR #6 in this session and had to recover from local reflog.

**Do**: After PR merge, sync the local branch via
`git fetch && git reset --hard origin/main && git push -f origin <branch>`.
ONLY after the PR has been merged.

**See also**: This session's commit history; the recovery is
documented in `ITERATION_REPORT.md` Phase 4.

### 3.3 MCP read returns HTML-escaped content

**Do**: always run `html.unescape()` on issue/comment bodies
returned by `mcp__github__*_read` tools before JSON-parsing or
literal-string matching.

**Why**: MCP returns `&#34;` instead of `"`, `&amp;` instead of `&`,
etc. Strict parsers fail silently on this. Idempotent on REST-
returned content, so safe to apply unconditionally.

**See also**: `retrospective/github-mcp-tips/SPEC.md` §2.1

### 3.4 MCP-posted comments get an auto-trailer

**Do**: parse posted comments with `JSONDecoder().raw_decode()`
which extracts the longest JSON-object prefix. Tolerates trailing
prose.

**Why**: `mcp__github__add_issue_comment` unconditionally appends
`\n\n---\n_Generated by [Claude Code](https://claude.ai/code)_` to
every comment body. Strict `json.loads()` fails.

```python
import html, json
def parse_lenient(body: str):
    body = html.unescape(body).lstrip()
    try:
        parsed, _ = json.JSONDecoder().raw_decode(body)
        return parsed
    except (json.JSONDecodeError, ValueError):
        return None
```

**See also**: `retrospective/github-mcp-tips/SPEC.md` §2.2

### 3.5 Don't lock issues that workflows need to write to

**Don't**: gate workflows on `github.event.issue.locked == true`.

**Why**: Locked issues refuse comments from `GITHUB_TOKEN` (the
github-actions[bot] identity), even with `permissions: issues:
write`. The workflow's own writes get 403. This was the highest-
impact bug discovered in this session.

**Do**: filter on label + author in the `if:` clause. If you need
a tamper-prevention seal, lock the issue at close, not at
creation.

**See also**: `retrospective/github-mcp-tips/SPEC.md` §2.7,
ITERATION_REPORT.md Phase 4 bug #6.

### 3.6 Source `agent_login` from the actual MCP user

**Don't**: hardcode `agent_login` to a placeholder like `"my-bot"`.

**Why**: The MCP server authenticates as the human user
(e.g. `jonathanmanton`). Workflow `if:` filters comparing to
`'my-bot'` will never match.

**Do**: At session start, call `mcp__github__get_me` and persist
the returned `login` into the project's config. Reference that
config from workflow YAMLs.

**See also**: `retrospective/github-mcp-tips/SPEC.md` §2.8

### 3.7 Subagents must commit locally before stopping

**Do**: in subagent briefs, instruct: "if you commit local changes
during your work, push them before stopping. The dispatcher can
pick up local-but-unpushed work."

**Why**: Subagents can run out of usage mid-task. If they've
committed locally, the dispatcher can recover. Without a local
commit, the work is lost.

**See also**: `retrospective/subagent-prompting/SPEC.md` §6

### 3.8 Subagent briefs must specify exact deliverables

**Do**: every subagent brief ends with: "When done: (1) commit
+ push to <branch> with message X, (2) open PR via mcp tools with
title Y, (3) report PR# + test count + caveats in under N words."

**Why**: Without exact deliverables, results vary wildly across
subagents. Reports balloon, branches don't get pushed, PRs don't
get opened.

**See also**: `retrospective/subagent-prompting/SPEC.md` §2

### 3.9 Stub `__main__` blocks should print env requirements

**Do**: workflow Python scripts have a `__main__` block that lists
required env vars and exits non-zero if any are missing. Print to
stderr.

**Why**: makes "how do I invoke this" self-documenting. Catches
config drift between YAML and code.

```python
def main() -> int:
    required = ["ISSUE_NUMBER", "COMMENT_ID", "GITHUB_TOKEN", "GITHUB_REPOSITORY"]
    print(f"handler: required env vars: {', '.join(required)}", file=sys.stderr)
    missing = [n for n in required if not os.environ.get(n)]
    if missing:
        print(f"handler: missing env vars: {missing}", file=sys.stderr)
        return 1
    # ... rest
```

**See also**: `retrospective/github-mcp-tips/SPEC.md` §2.7

### 3.10 Diagnostic comments are first-class

**Do**: workflow scripts post a self-diagnostic comment on
uncaught exception (with `traceback.format_exc()` in a fenced
code block) on the originating issue. Wrap in its own try/except.

**Why**: MCP-only operators have no other way to read CI logs.
This pattern saves hours of debug spelunking.

**See also**: `retrospective/live-debug-from-mcp-only/SPEC.md`

### 3.11 Forensic by default, aggressive on success

**Default**: forensic preservation of artifacts. Failed scenarios
leave everything for inspection.

**Auto-cleanup**: bake into `close_on_merge.py` (or equivalent) —
when a PR for an agent issue is merged, sweep the head branch
+ all `<feature>--sub-*` siblings.

**Defensive**: never auto-delete `main`, default branch, or orphan
log branches like `_agent_runs`. Only operate within an explicit
agent namespace prefix.

**See also**: `retrospective/forensic-vs-aggressive-cleanup/SPEC.md`

### 3.12 Polling intervals for remote APIs

**Do**: 30s minimum interval when polling remote APIs (GitHub,
similar). Cap total wait at 1 hour for any single operation.

**Why**: Faster polling hits rate limits and adds noise to the
notification stream without finding state changes faster.

**See also**: `retrospective/polling-without-sleep-in-restricted-sandbox/SPEC.md`

### 3.13 Don't expect to read CI logs

**Note**: `mcp__github__*` tools do not expose workflow run logs.
WebFetch on `/actions/runs/<id>` hits an auth wall even on public
repos.

**Do**: bake diagnostic comment patterns into workflows from the
start (don't wait until you need them).

**See also**: `retrospective/live-debug-from-mcp-only/SPEC.md`

### 3.14 Run live scenarios after unit tests pass

**Do**: don't ship based on unit-test green. Run live scenarios
that exercise architecturally-distinct code paths. Spec defects
and transport quirks only surface live.

**See also**: `retrospective/spec-vs-implementation-gap-discovery/SPEC.md`

### 3.15 Spec amendments inline

**Do**: when live execution discovers a spec problem, amend the
spec inline with a "Real-world correction (discovered <when> via
<how>)" paragraph at the relevant section.

**Why**: discoveries that don't make it back into the spec are
lost. Inline amendments survive context turnover.

**See also**: `retrospective/spec-vs-implementation-gap-discovery/SPEC.md` §6

## 4. Suggested template

When generating an actual `AGENTS.md` for a new project, organize
the rules above by category:

```markdown
# AGENTS.md

This file documents conventions for AI agents working on this
project. Most rules are grounded in real failures from a previous
session — see `retrospective/` for full provenance.

## Working with the GitHub MCP server
- §3.3 HTML-escape on read
- §3.4 Trailer tolerance on parse
- §3.5 Don't lock issues mid-workflow
- §3.6 Source agent_login from get_me
- §3.13 No CI log access from MCP

## Branch + PR coordination
- §3.1 Use `--sub-` separator
- §3.2 Don't force-push during open PR
- §3.11 Forensic by default; auto-cleanup on merge

## Subagent dispatch
- §3.7 Commit before stopping
- §3.8 Specify exact deliverables

## Workflow scripts
- §3.9 `__main__` prints env requirements
- §3.10 Diagnostic comments are first-class
- §3.12 Polling intervals

## Spec discipline
- §3.14 Live scenarios after unit tests
- §3.15 Inline spec amendments
```

## 5. Project-specific additions

Beyond the universal rules above, each project may add:

- The project's specific branch naming convention
- The project's specific scenario-id format
- The project's test command + coverage target
- The project's minimum coverage gates
- Any custom MCP tools or harnesses

Keep these in their own section so the universal rules stay
identifiable.

## 6. Anti-patterns for AGENTS.md itself

- **Too long.** >200 lines means agents won't read it all.
- **Vague advice.** Each rule needs a concrete failure mode.
- **No "see also" links.** Without them, the rules are floors with
  no way down to depth.
- **Mixing project-specific and universal rules.** Separate them so
  the universal ones can be reused.

## 7. Test plan (when implemented)

- Verify each rule has a real failure example in
  ITERATION_REPORT.md or session evidence.
- Verify each "see also" link points to a real spec section.
- Run on a fresh project: agent reads AGENTS.md and demonstrably
  follows the conventions.

## 8. Living document

New rules get added as new sessions discover new failures.
Existing rules get refined as patterns evolve. The file is never
"done."
