# Step 1: review the GitHub POC before consulting the alternatives

The GitHub POC was in progress when this decision package was written.
The point of starting here is to learn from what was actually built,
not just what `SPEC.md` describes.

## Why this matters

Designs make assumptions; implementations discover constraints. The
Bitbucket adapter inherits the constraints discovered by the GitHub
POC, not the assumptions in the original draft. If the POC discovered
that, say, locking issues at creation breaks the workflow's terminal
write — the Bitbucket adapter must not repeat that mistake. (Spoiler:
this one is already documented in `SPEC.md` §3 "Real-world correction"
because it was discovered live.)

## Files to read

In order:

1. `README.md` — current high-level pitch.
2. `SPEC.md` — current spec, including the "Real-world correction"
   sections that flag where the POC diverged from the original draft.
3. `ITERATION_REPORT.md` — iteration journal if present.
4. `harness/RUNS.md` and `harness/runs/*.md` — concrete records of live
   POC runs, including issue numbers, comment IDs, and workflow runs.
5. `git log --oneline` from the most recent ~30 commits — the commit
   messages narrate the implementation journey.
6. `.agent/scripts/handler.py` — the workflow handler. Note the
   `try/finally` that produces a terminal envelope even on uncaught
   exceptions, the JSON-prefix-tolerant body parser, and the HTML
   unescape.
7. `.agent/scripts/lock_and_sweep.py` — note that locking happens at
   close, not creation (per the §3 correction).
8. `.agent/scripts/close_on_merge.py` — note how it correlates merged
   PRs back to issues.
9. `.github/workflows/*.yml` — note the `if:` filter on
   `batch-job-handler.yml` (label + comment author) and how it
   provides the public-repo safety property.
10. `.agent/config.json` — current tunables.
11. `.agent/schemas/**` — current schemas.
12. `skills/batch-job/*.py` and `skills/task-dag/*.py` — agent-side
    code; these are the operations the Bitbucket adapter must support.

## Things to flag explicitly

For each of the following, write a one-line note in your review:

- [ ] Was the lock-at-close fix shipped in the POC, or is it still
      open? (Commit `11ddda3` suggests yes; verify.)
- [ ] Did the POC have to add an HTML-unescape step on issue body
      parsing to handle MCP responses?
      (Look in `.agent/scripts/common.py` for `html.unescape`.)
- [ ] Did the POC have to add tolerant JSON parsing for MCP-injected
      trailers like `_Generated by ..._`?
      (Look for "longest JSON-object prefix" logic.)
- [ ] What is the actual `if:` filter on the workflow today? (Author +
      label, or has it diverged?)
- [ ] Were any new failure modes discovered that aren't in `SPEC.md`
      §12?
- [ ] Did `_agent_runs` orphan-branch commits hit any non-fast-forward
      issues that required retry? Is the retry implemented?
- [ ] What's the actual workflow-pickup latency seen in
      `harness/runs/*.md`? (Affects the
      `runner_pickup_timeout_seconds` value the Bitbucket adapter
      should pick.)
- [ ] Were any commands actually implemented and exercised end-to-end
      (e.g. `run-tests`, `echo`)? What did their `summary` shapes look
      like?
- [ ] What MCP server was used for the live runs (`vendor/version`)?
      Did it expose all the MCP tools the SPEC §16 compatibility
      section claims are needed?
- [ ] Were there any agent harness limitations discovered (e.g. tool
      rate limiting, comment body size limits, list-comments
      pagination)?

## Output

Write a markdown section titled "POC deltas" with:

```
## POC deltas (as of <commit-sha>)

### Confirmed in POC
- ...

### Diverges from SPEC.md
- ...

### Open / unverified
- ...
```

Reference commits by short SHA. Quote `harness/runs/` files where
relevant. Do not editorialise — just record.

## What you do with this

The "POC deltas" section becomes part of your final `DECISION.md`. Any
alternative in `ALTERNATIVES.md` that conflicts with a confirmed POC
delta should be flagged when you walk through the decision questions
with the user. For example, if the POC discovered that some MCP server
silently rate-limits comment listings, the Bitbucket adapter needs to
plan for the same on its substrate of choice.
